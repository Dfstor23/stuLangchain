LangChain 是一个构建 LLM（大语言模型）应用的框架，它的核心能力包括：
1. Prompt 模板化：把用户输入填入预设模板，生成标准化的模型输入。
2. 调用模型：统一接口调用 OpenAI、DeepSeek、文心一言等多种大模型。
3. 检索增强生成（RAG）：先从外部知识库检索相关文档片段，再把片段和问题一起送给模型，让回答有据可查。
4. 工具调用和 Agent 编排：让模型自主决定何时调用搜索、计算器、数据库等外部工具。

Prompt 模板的作用：
Prompt 模板用于把用户问题、检索到的上下文和固定指令组合成标准化输入。
它可以约束模型的回答范围，例如要求模型只能根据资料回答，资料不足时明确说明不知道。
在 RAG 中，Prompt 模板通常包含 context 和 question 两个变量，其中 context 是检索到的文档片段，
question 是用户原始问题。

RAG 的关键步骤：
第一步，加载文档，支持 txt、pdf、网页等多种格式。
第二步，把文档切成小块，每块通常 200 到 500 个字符，块之间保留一定重叠防止信息断裂。
第三步，用 Embedding 模型把每个文本块转成向量，存入向量数据库如 Chroma、FAISS。
第四步，用户提问时，先把问题转成向量，在向量库中检索最相似的几个文本块。
第五步，把检索到的文本块作为上下文，和用户问题一起填入 Prompt 模板，交给大模型生成回答。

Document.metadata 的作用：
Document.metadata 用来保存文档片段的附加信息，例如 source 表示片段来自哪个文件，chunk_id 表示这是第几个文本块。
metadata 不直接作为正文回答用户问题，但它可以帮助系统追踪答案来源、打印命中片段、做来源引用和调试检索结果。
在 RAG 项目中，metadata 能让开发者知道模型回答依据的是哪份资料、哪个片段，方便检查回答是否可靠。

为什么需要 RAG？
大模型的训练数据有截止日期，无法回答训练后发生的事件。同时大模型可能编造事实（幻觉）。RAG 通过注入外部知识，
让回答基于真实文档，减少幻觉，并且可以标注来源，方便核实。

LangGraph 是什么？
LangGraph 是 LangChain 生态中的底层编排框架，主要用于构建长时间运行、有状态的 Agent 工作流。
它可以把 Agent 的执行过程组织成图结构，并支持状态管理、持久化执行、人工介入和复杂流程控制。
和普通的一次性问答链相比，LangGraph 更适合需要多步骤决策、工具调用、循环执行或中途暂停恢复的 Agent 应用。
