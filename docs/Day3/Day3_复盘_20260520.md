## questions.md：检查题与点评

### 第 1 组：命令行 RAG Demo 主流程

已掌握：

- 先看命中片段，再看模型回答，可以判断回答是否基于资料。
- `strip()` 可以去掉输入两端的空格、换行和制表符，减少空问题。
- `TOP_K` 变小会减少命中片段，可能导致回答不完整。
- 缺少 `DEEPSEEK_API_KEY` 优先属于 API 配置问题。

纠正点：

```python
if question.lower() == "exit" or "quit":
```

这个写法是错的，因为 `"quit"` 本身永远为真。正确写法是：

```python
if question.lower() in ["exit", "quit", "q"]:
```

### 第 2 组：异常提示

已掌握：

- 没有命中片段时继续让模型回答，容易产生幻觉。
- `continue` 会跳过本轮循环后面的代码，进入下一轮输入。
- `TOP_K=4` 不代表一定返回 4 个片段，如果知识库片段不足，返回数量会更少。

纠正点：

- “缺少 API Key”主要是 API 配置问题。
- 只有在代码没有正确调用 `load_dotenv()`、变量名写错或路径不对时，才可能同时是代码问题。

### 第 3 组：来源引用

已掌握：

- 来源引用能说明模型回答有据可循。
- `chunk_id` 是我们写入 `Document.metadata` 的片段编号，不是 Chroma 自动生成的内部 ID。
- `TOP_K=4` 时，来源列表最多打印 4 行。
- 如果 `chunk_id=None`，通常说明构建 `Document` 时没有写入对应 metadata。

小改造：

把完整路径改成只显示文件名：

```python
source = Path(doc.metadata.get("source")).name
```

### 第 4 组：README 运行说明

已掌握：

- README 里的示例问题可以帮助使用者知道如何提问。
- 使用 `.venv\Scripts\python.exe` 是为了确保运行在安装了 LangChain 等依赖的虚拟环境中。
- README 要尽量和实际代码一致。
- 缺少 `DEEPSEEK_API_KEY` 时，要检查 `.env` 中变量是否存在、变量名是否正确、key 是否有效。

纠正点：

- 如果 `TOP_K` 从 4 改成 2，README 不一定必须改。只有 README 明确写了具体返回数量时才必须同步修改。

## blockers.md：卡点与报错

### 1. `DEEPSEEK_API_KEY` 没有读到

现象：

```text
启动失败：缺少 DEEPSEEK_API_KEY
请先在项目根目录的 .env 文件中配置 DEEPSEEK_API_KEY。
```

类型：API 配置问题。

排查顺序：

1. 检查项目根目录是否有 `.env`。
2. 检查变量名是否为 `DEEPSEEK_API_KEY`。
3. 检查 key 是否为空或过期。
4. 检查代码是否调用了 `load_dotenv()`。

### 2. 退出条件写法错误

错误写法：

```python
if question.lower() == "exit" or "quit":
```

原因：

`"quit"` 是非空字符串，在 Python 中会被当成真值，导致条件永远成立。

正确写法：

```python
if question.lower() in ["exit", "quit", "q"]:
```

### 3. source 完整路径太长

问题：

来源引用里显示完整路径，不够简洁。

最小修改：

```python
source = Path(doc.metadata.get("source")).name
```

效果：

```text
source=intro.txt
```
