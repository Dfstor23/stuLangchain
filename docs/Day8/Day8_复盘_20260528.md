- 在 `day2.py` 的 Prompt 中加了一句：`请尽量完整地列出资料中提到的所有相关要点，不要遗漏。`
- 复测后回答从 1 个要点变成 4 个要点，"标注来源"不再遗漏。

## 2. 今天掌握的知识点

- 测试问题集分为三类：知识库内、知识库边界、知识库外。
- 三类问题分别验证：正常回答能力、部分覆盖表现、拒答能力。
- RAG 有两种优化手段：改知识库（解决信息缺失）、改 Prompt（解决信息提取不完整）。
- Prompt 作用于生成环节，不影响检索环节。
- 向量检索永远会返回 top_k 个结果，拒答能力靠 Prompt 约束。
- README 面向面试官和 GitHub 访客，功能和技术栈放前面，学习记录放末尾。
- 项目结构只展示核心文件，避免冗杂。

## 3. 需要记住的补强点

- 排查问题按 API 配置 → 检索结果 → Prompt 拼接的顺序。
- 知识库补强和 Prompt 优化都是闭环：改完必须复测。
- 拒答措辞需要同时改 Prompt（day2.py）和前端提示（streamlit_rag_demo.py）。
- Streamlit 支持热更新，改代码后刷新或点 Rerun 即可，不需要重启。
- `ModuleNotFoundError` 几乎都是环境问题（未激活虚拟环境），不是代码问题。
- 面试回答参数选择时，要体现做过对比实验，而不是拍脑袋选的。

## 4. 面试表达

### RAG 质量评估

> 我建立了一个分类测试问题集，把问题分为知识库内、知识库边界和知识库外三类，分别验证 RAG 系统的正常回答能力、部分覆盖时的表现、以及资料不足时的拒答能力。在测试中发现边界问题回答偏简短，通过优化 Prompt 要求模型完整提取资料要点，回答质量显著提升，形成了"发现问题 → 分析原因 → 最小修改 → 复测验证"的质量优化闭环。

### 项目整体介绍

> 我做了一个基于本地知识库的 RAG 问答系统，用户输入问题后，系统从知识库中检索相关片段，结合大模型生成回答。技术栈用了 LangChain 构建 RAG 链路、Chroma 做向量存储、HuggingFace 本地 Embedding 模型、DeepSeek API 做生成、Streamlit 做前端展示。项目亮点是建立了分类测试问题集来系统评估回答质量，并且通过 Prompt 优化和知识库补强两种手段形成了可复现的质量优化闭环。

### 参数选择

> 我用固定问题集做了参数对比实验：测试了不同 chunk_size（80/100/150/200/300/400）和不同 chunk_overlap（0/30/80），观察命中片段和回答质量的变化。当前参数在小知识库场景下，知识库内问题回答完整，边界问题也能正确回答，知识库外问题能拒答，综合表现最稳定。

## 5. 检查题记录

### Day8 第 1 步检查题

1. 测试问题集分三类的原因：只测知识库内问题无法验证拒答能力，分类测试才能全面评估。✅
2. 知识库外问题为什么检索到 LangGraph：top_k=4 决定一定返回 4 个片段，且 LangGraph/LangSmith 向量相似度较近。✅
3. top_k 增大能否改善回答遗漏：不能，因为片段已命中，问题在模型生成环节，应改 Prompt。⚠️（原因分析需更精准）
