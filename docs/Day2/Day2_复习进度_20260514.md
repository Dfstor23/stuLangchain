## 第 3 步：Chroma 向量库完整链路 ✅

### 本步目标

跑通：加载文档 → 文本切分 → embedding → 存入 Chroma → 用户问题检索 → 打印命中片段。

### 已掌握

- Chroma 保存的是知识库文本 chunk 的向量。
- 用户问题不会提前存入 Chroma，而是在提问时临时转成向量，再去 Chroma 中检索相似文本。
- `Document.page_content` 保存文本内容。
- `Document.metadata` 保存来源信息，例如 `source` 和 `chunk_id`。
- `chunk_id` 是我们自己写入 metadata 的片段编号，不等于 Chroma 内部数据库 ID。
- `k=3` 表示返回最相关的 3 个片段；如果只想返回 1 个片段，应改成 `k=1`。

### 已纠正的问题

- 一开始误以为用户问题也会存入 Chroma，已纠正为：用户问题只在检索时临时转向量。
- 一开始误以为 `k=3` 只返回一个片段，已纠正为：`k=3` 返回 3 个片段。

## 第 4 步：完整 RAG + `k` 参数对比实验 ✅

### 本步目标

把第 3 步检索到的文档片段和用户问题一起传给大模型，让模型基于检索资料生成回答，并对比 `k=1` 和 `k=3` 的效果。

### 已掌握

- 完整 RAG 比单纯检索多了“把检索片段和用户问题一起交给 LLM 生成回答”的步骤。
- `answer = rag_chain.invoke({"context": context, "question": question})` 中：
  - `context` 是检索到的文本片段拼接结果。
  - `question` 是用户原始提问。
- `k=1` 表示只根据 1 个命中片段回答，资料可能不够完整。
- `k=3` 表示根据 3 个命中片段回答，通常回答更完整。

### 参数实验结论

- `k=1`：上下文较少，回答可能有缺漏。
- `k=3`：上下文更充分，回答通常更完整。
- `k` 不是越大越好，后续需要用固定问题集对比检索质量和回答质量。

### 报错排查记录

如果出现 `缺少 DEEPSEEK_API_KEY`，优先检查：

- `.env` 文件中是否存在有效的 `DEEPSEEK_API_KEY`。
- 代码是否调用了 `load_dotenv()`。
- 当前运行环境是否能读取到该环境变量。

## Day2 阶段总结

### 今天掌握了什么

- LCEL 中 `RunnablePassthrough()` 的作用是透传用户原始问题。
- LCEL 字典结构会在链中并行生成多个字段，例如同时生成 `context` 和 `question`。
- 关键词检索无法处理同义表达，embedding 检索可以根据语义相似度找到相关片段。
- Chroma 用来保存知识库文本向量，并支持按用户问题检索相似片段。
- 完整 RAG 的基本流程是：问题 → 检索 → 上下文注入 Prompt → LLM 回答。

### 当前卡点与已修正点

- `embed_query()` 返回的是 `List[float]`，不是 int。
- `normalize_embeddings=True` 后，点积结果才可以按余弦相似度理解。
- 用户问题不会提前存入 Chroma。
- `k=3` 返回 3 个片段，不是 1 个片段。
- `DEEPSEEK_API_KEY` 问题应优先检查 `.env` 和 `load_dotenv()`。

## RAG 调参复习：`top_k` / `chunk_size` / `chunk_overlap` ✅

### 第 1 个实验：`top_k` 对比

固定问题：`RAG 的关键步骤是什么？`

固定参数：

- `chunk_size=150`
- `chunk_overlap=30`

对比参数：

- `k=1`
- `k=3`
- `k=5`

实验结论：

- `k=1`：返回 1 个片段，命中较精准，但可能缺少完整信息。
- `k=3`：返回 3 个片段，通常更完整，是当前小文档里较平衡的选择。
- `k=5`：返回 5 个片段，信息更多，但容易混入弱相关或无关片段。

已掌握：

- `top_k` 控制检索返回的片段数量。
- `k` 不是越大越好，过大可能让上下文变杂，影响回答准确性。
- 向量库应尽量只构建一次，再用不同 `k` 做检索对比，避免重复删除 Chroma 数据库导致 Windows 文件占用问题。

### 第 2 个实验：`chunk_size` 对比

固定问题：`RAG 的关键步骤是什么？`

固定参数：

- `chunk_overlap=30`
- `k=3`

对比参数：

- `chunk_size=80`
- `chunk_size=150`
- `chunk_size=300`
- 小改造补充：`chunk_size=100 / 200 / 400`

实验结论：

- `chunk_size=80`：切块数量更多，片段较碎，容易导致回答信息不完整。
- `chunk_size=150`：切块数量适中，能覆盖部分关键步骤。
- `chunk_size=300`：片段更完整，但有混入较泛内容的风险。
- 小改造中，`chunk_size=200` 对当前问题较合适：切成 5 块，命中片段较完整。

已掌握：

- `chunk_size` 控制每个文本块的大小。
- `chunk_size` 太小，信息容易被切碎。
- `chunk_size` 太大，单个片段更完整，但可能包含更多无关内容。
- 调参时要使用控制变量法，一次只改一个参数。

### 第 3 个实验：`chunk_overlap` 对比

固定问题：`RAG 的关键步骤是什么？`

固定参数：

- `chunk_size=200`
- `k=3`

对比参数：

- `chunk_overlap=0`
- `chunk_overlap=30`
- `chunk_overlap=80`

实验结论：

- `chunk_overlap=0`：当前小文档下已经能命中主要内容，但第 5 步在另一个片段中。
- `chunk_overlap=30`：和 `overlap=0` 基本接近，变化不明显。
- `chunk_overlap=80`：命中片段中可能出现更多弱相关内容。

小改造问题：`Prompt 模板有什么作用？`

- `overlap=0` 和 `overlap=30`：命中结果基本相同。
- `overlap=80`：会带入一些无用文本。

已掌握：

- `chunk_overlap` 控制相邻 chunk 之间重复的字符数量。
- overlap 可以减少信息断裂，但不是越大越好。
- overlap 太大可能增加重复内容，甚至让检索上下文变杂。
- 在短文档里，`chunk_overlap` 的效果可能不明显；长文档中会更重要。

### 当前调参结论

针对当前 `data/intro.txt` 这个小知识库，较合理的初始参数是：

```text
chunk_size = 200
chunk_overlap = 30
top_k = 3
```

这不是最终最优参数，只是当前小文档和当前问题集下的阶段性选择。后续应继续用固定问题集记录命中片段、回答完整度和无关内容比例。
