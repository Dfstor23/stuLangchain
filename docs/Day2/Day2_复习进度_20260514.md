## 第 3 步：Chroma 向量库完整链路 ✅

### 本步目标

跑通：加载文档 → 文本切分 → embedding → 存入 Chroma → 用户问题检索 → 打印命中片段。

### 已掌握

- Chroma 保存的是知识库文本 chunk 的向量。
- 用户问题不会提前存入 Chroma，而是在提问时临时转成向量，再去 Chroma 中检索相似文本。
- `Document.page_content` 保存文本内容。
- `Document.metadata` 保存来源信息，例如 `source` 和 `chunk_id`。
- `chunk_id` 是我们自己写入 metadata 的片段编号，不等于 Chroma 内部数据库 ID。
- `k=3` 表示返回最相关的 3 个片段；如果只想返回 1 个片段，应改成 `k=1`。

### 已纠正的问题

- 一开始误以为用户问题也会存入 Chroma，已纠正为：用户问题只在检索时临时转向量。
- 一开始误以为 `k=3` 只返回一个片段，已纠正为：`k=3` 返回 3 个片段。

## 第 4 步：完整 RAG + `k` 参数对比实验 ✅

### 本步目标

把第 3 步检索到的文档片段和用户问题一起传给大模型，让模型基于检索资料生成回答，并对比 `k=1` 和 `k=3` 的效果。

### 已掌握

- 完整 RAG 比单纯检索多了“把检索片段和用户问题一起交给 LLM 生成回答”的步骤。
- `answer = rag_chain.invoke({"context": context, "question": question})` 中：
  - `context` 是检索到的文本片段拼接结果。
  - `question` 是用户原始提问。
- `k=1` 表示只根据 1 个命中片段回答，资料可能不够完整。
- `k=3` 表示根据 3 个命中片段回答，通常回答更完整。

### 参数实验结论

- `k=1`：上下文较少，回答可能有缺漏。
- `k=3`：上下文更充分，回答通常更完整。
- `k` 不是越大越好，后续需要用固定问题集对比检索质量和回答质量。

### 报错排查记录

如果出现 `缺少 DEEPSEEK_API_KEY`，优先检查：

- `.env` 文件中是否存在有效的 `DEEPSEEK_API_KEY`。
- 代码是否调用了 `load_dotenv()`。
- 当前运行环境是否能读取到该环境变量。

## Day2 阶段总结

### 今天掌握了什么

- LCEL 中 `RunnablePassthrough()` 的作用是透传用户原始问题。
- LCEL 字典结构会在链中并行生成多个字段，例如同时生成 `context` 和 `question`。
- 关键词检索无法处理同义表达，embedding 检索可以根据语义相似度找到相关片段。
- Chroma 用来保存知识库文本向量，并支持按用户问题检索相似片段。
- 完整 RAG 的基本流程是：问题 → 检索 → 上下文注入 Prompt → LLM 回答。

### 当前卡点与已修正点

- `embed_query()` 返回的是 `List[float]`，不是 int。
- `normalize_embeddings=True` 后，点积结果才可以按余弦相似度理解。
- 用户问题不会提前存入 Chroma。
- `k=3` 返回 3 个片段，不是 1 个片段。
- `DEEPSEEK_API_KEY` 问题应优先检查 `.env` 和 `load_dotenv()`。

### 下一步

进入 RAG 质量优化：用固定问题集对比 `chunk_size`、`chunk_overlap`、`top_k` 对检索结果和回答质量的影响。
