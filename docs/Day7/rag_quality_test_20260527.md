# Day7 RAG 回答质量测试记录

## 测试目标

用 3 个固定问题观察当前 Streamlit RAG Demo 的检索和回答质量。

重点看三件事：

- 命中片段是否和问题相关。
- 模型回答是否基于命中片段。
- 回答是否遗漏知识库里的关键点。

## 当前参数

```text
chunk_size = 200
chunk_overlap = 30
top_k = 4
```

## 运行命令

```powershell
.\.venv\Scripts\python.exe -m streamlit run docs\Day4\streamlit_rag_demo.py
```

## 测试问题 1

问题：

```text
RAG 的关键步骤是什么？
```

命中片段记录：

```text
片段 1 | source=intro.txt | chunk_id=3

RAG 的关键步骤： 第一步，加载文档，支持 txt、pdf、网页等多种格式。 第二步，把文档切成小块，每块通常 200 到 500 个字符，块之间保留一定重叠防止信息断裂。 第三步，用 Embedding 模型把每个文本块转成向量，存入向量数据库如 Chroma、FAISS。 第四步，用户提问时，先把问题转成向量，在向量库中检索最相似的几个文本块。

片段 2 | source=intro.txt | chunk_id=7

为什么需要 RAG？ 大模型的训练数据有截止日期，无法回答训练后发生的事件。同时大模型可能编造事实（幻觉）。RAG 通过注入外部知识， 让回答基于真实文档，减少幻觉，并且可以标注来源，方便核实。

片段 3 | source=intro.txt | chunk_id=6

在 RAG 项目中，metadata 能让开发者知道模型回答依据的是哪份资料、哪个片段，方便检查回答是否可靠。

片段 4 | source=intro.txt | chunk_id=4

第五步，把检索到的文本块作为上下文，和用户问题一起填入 Prompt 模板，交给大模型生成回答。
```

模型回答：

```text
根据资料，RAG 的关键步骤包括：

1.加载文档（支持 txt、pdf、网页等多种格式）。
2.把文档切成小块（每块通常 200 到 500 个字符，块之间保留一定重叠）。
3.用 Embedding 模型把每个文本块转成向量，存入向量数据库（如 Chroma、FAISS）。
4.用户提问时，先把问题转成向量，在向量库中检索最相似的几个文本块。
5.把检索到的文本块作为上下文，和用户问题一起填入 Prompt 模板，交给大模型生成回答。
```

简单评价：

```text
回答覆盖加载文档，切块大小较适合(包含了大部分文档资料)，向量库检索较准确、Prompt 注入和模型回答精确，
Enbedding向量化不知道如何评价。
```

## 测试问题 3

问题：

```text
Document.metadata 可以用来做什么？
```

```text
片段 1 | source=intro.txt | chunk_id=5

Document.metadata 的作用： Document.metadata 用来保存文档片段的附加信息，例如 source 表示片段来自哪个文件，chunk_id 表示这是第几个文本块。 metadata 不直接作为正文回答用户问题，但它可以帮助系统追踪答案来源、打印命中片段、做来源引用和调试检索结果。

片段 2 | source=intro.txt | chunk_id=6

在 RAG 项目中，metadata 能让开发者知道模型回答依据的是哪份资料、哪个片段，方便检查回答是否可靠。

片段 3 | source=intro.txt | chunk_id=7

为什么需要 RAG？ 大模型的训练数据有截止日期，无法回答训练后发生的事件。同时大模型可能编造事实（幻觉）。RAG 通过注入外部知识， 让回答基于真实文档，减少幻觉，并且可以标注来源，方便核实。

片段 4 | source=intro.txt | chunk_id=1

工具调用和 Agent 编排：让模型自主决定何时调用搜索、计算器、数据库等外部工具。
```

简单评价：

```text
回答说明 source、chunk_id、来源引用和调试检索结果。
```

## 带教点评

本次 3 个问题都属于知识库内问题，整体表现较好：

- “RAG 的关键步骤是什么？”命中了关键步骤片段，并补到了第 5 步。
- “Prompt 模板在 RAG 中有什么作用？”命中了 Prompt 模板片段，回答覆盖 context、question 和资料不足提示。
- “Document.metadata 可以用来做什么？”命中了 metadata 片段，回答覆盖 source、chunk_id、来源引用和调试检索结果。

需要修正一个判断点：

```text
Embedding 本身不用直接评价数学效果。
在当前阶段，只需要观察“命中片段是否和问题相关”。
如果相关，说明向量化 + 检索这一步基本有效。
```

更推荐的下一步最小优化方向：

```text
新增 1 个知识库外问题，测试页面是否能正确提示“资料中没有足够信息。”。
```

原因：

- 当前 3 个问题都能从 `data/intro.txt` 找到答案。
- 还没有验证“资料不足时拒答”的效果。
- 这比直接补充知识库内容更适合 Day7 的 RAG 质量评估目标。

## Day7 最终结论

```text
当前 RAG Demo 对知识库内问题能给出较准确回答；
对知识库外问题能提示资料不足，具备初步降低幻觉风险的能力。
下一步可以考虑补充知识库内容，或开始做更正式的测试问题集。
```
